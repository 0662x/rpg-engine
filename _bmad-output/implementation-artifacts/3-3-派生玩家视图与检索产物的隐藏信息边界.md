# Story 3.3: 派生玩家视图与检索产物的隐藏信息边界

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## 用户故事

作为玩家，
我希望 cards、snapshots、FTS、onboarding 和其他派生玩家视图也不包含 hidden / GM-only 信息，
从而避免隐藏内容通过 read model 或搜索产物泄漏。

## 验收标准

1. 给定 hidden 或 GM-only material 存在于 Save facts、events、memory、cards source 或 projection input，当 onboarding、snapshots、cards、FTS 或 player-facing search artifacts 被生成或刷新时，player-visible artifacts 不包含 hidden material，并且 projection/search 的派生缓存不能成为 hidden 泄漏路径。
2. 给定 visibility 相关实现发生变化，当 focused visibility tests 运行时，tests 必须分别覆盖 player view、ordinary query、FTS、scene output、cards、snapshots 和 AI prompt boundaries，并且 failures 能指出泄漏的 item type 与 artifact type。

### 当前实现状态

- `rpg_engine/db.py` 的 `rebuild_fts()` 已按 player visibility 过滤 non-archived entities，并对 title/body/tags 调用 hidden ref redaction；`resolve_entity(..., view="player")` 会在 exact/name/alias/token/body/FTS fallback 阶段加 visibility SQL。
- `rpg_engine/render.py` 的 `render_scene()`、`render_current_snapshot()`、`render_current_snapshot_json()` 已支持 `view`，默认 `PLAYER_VIEW`，并在 scene/present/carried/clocks/current location 查询中复用 visibility SQL。
- `rpg_engine/cards.py` 的 `write_cards()` 默认 `index_view=PLAYER_VIEW`，读取 cards source 时会用 visibility SQL；`render_card()` 和 index rendering 已使用 hidden id redaction 做二层防护。
- `rpg_engine/projection_service.py` 的 `_write_projection()` 对 `search` 调 `rebuild_fts()`，对 `snapshots` 调 `write_current_snapshot()` / `write_current_snapshot_json()`，对 `cards` 调 `write_cards()`；这些 projection refresh 只写 derived artifacts 和 `projection_state` evidence。
- `rpg_engine/save_validation.py` 已对 snapshots、cards、FTS 执行 hidden entity ref 扫描和 stale generated card 检测；本 story 应补齐行为级 regression，避免只验证 artifact 存在。
- `rpg_engine/save_manager.py` 的 `start_or_continue()` 使用 `GMRuntime.query("scene", view="player")` 构造 onboarding scene；`render_onboarding_text()` 从 scene text 和 save record 摘要出玩家提示。

### 架构合规要求

- 本 story 触碰 visibility、projection/read model、FTS/search、onboarding 和 player-facing artifacts，属于高风险 hidden-content 边界；必须先写 focused tests，再实现最小改动。
- Player-facing artifacts 必须默认 `player` view；GM / maintenance reads 必须通过显式 view 或 maintenance surface，不得由 projection artifact 默认暴露。
- Projection refresh、repair、cards、snapshots、FTS 和 onboarding 都不能写 gameplay facts、确认 pending action、修改 turn delta、提升 profile 或授予 hidden permission。
- Render redaction 是 defense-in-depth。核心要求是在 projection/search/source collection 阶段排除 hidden / GM-only material，并用 artifact 内容扫描证明。
- 不新增运行时依赖，不引入新的 search engine，不重写 projection service；优先复用 `visibility.py` SQL helpers、`redaction.py` hidden-ref helpers、`ProjectionService` 和现有 tests。
