# Story 1.9: 库存消耗语义提交门

Status: done

Completion note: routine 单物品库存消耗提交门、临时 Save 回归、7 轮三路审查、最终 required gates 与 Test Automation/Review/Trace/Gate 证据均已完成。

## 用户故事

作为长期存档的玩家主机，
我希望声明单物品库存消耗的结构化 delta 在提交前核验实时库存、数量算术和 payload/upsert 对齐，
从而使库存不足、过期或畸形的消耗不能成为 SQLite 事实。

## 验收标准

1. 给定一个结构化事件通过 `consumed_item_id`、`before_quantity`、`consumed_quantity` 和 `after_quantity` 声明单物品库存消耗，当 `player_turn_commit` validation 运行时，当前 SQLite 数量必须等于 `before_quantity`；`consumed_quantity` 必须为有限正数，`after_quantity = before_quantity - consumed_quantity` 且结果不得小于零；payload 的物品、单位和数量必须与唯一匹配的 `upsert_entities` 物品更新一致。
2. 给定消耗声明存在库存不足、过期 `before_quantity`、payload/upsert 物品不一致、缺失或重复更新、非法数量、单位不一致或算术不一致，当 validation 或 commit 被调用时，提交必须在任何持久化之前被拒绝，并返回稳定、可断言的消耗校验错误；SQLite、库存、turn、event 与 `events.jsonl` 均保持不变。
3. 给定一个合法的单物品消耗 delta，当它通过已批准的 `TurnProposal` 提交时，只扣减声明的数量并保留物品的 unit、quality、properties、durability、owner、location、status、visibility 等 metadata；只产生预期的 turn 与 event，不修改其他库存。
4. 给定现有 craft、combat 或其他 action 使用不同的领域 payload contract，当本 Story 的校验加入既有 validation pipeline 时，其既有领域 validator 和合法提交行为不得退化；本 Story 不新增自然语言意图识别、外部或内部 AI 权威、一等 consumption action、fuzzy quantity strategy、第三方依赖或测试专用 production API。
5. 给定 focused 与回归测试运行，当测试需要执行写入时，所有写入只针对独立 temporary Save；source Campaign、formal Save、正式 registry 与 `data/game.sqlite` 事实源保持不变。

### 当前实现与首选落点

- `run_validation_pipeline()` 已在 schema、capability、resolver request/resolution 之后调用 `validate_resolver_contract_stage()`；该 stage 使用 Runtime 当前注入的 canonical `ActionResolverRegistry` 并把 resolver errors 转成 blocked validation evidence。不要新建并行 pipeline stage。[Source: `rpg_engine/validation_pipeline.py`]
- CON fixture 的 action/intent 是 `routine`。`ROUTINE_RESOLVER.validate_delta` 当前绑定 `validate_routine_delta()`，后者只检查 request、location 与 audit event，未读取 item 的 live quantity，也未核验 consumption payload/upsert。[Source: `rpg_engine/actions/routine.py`; `tests/automation_support/domain_deltas.py`]
- `save.py` 在 transaction 内依次写 turn/event，再把 `upsert_entities` 交给 `db.upsert_entity()`；后者会覆盖 entity/item 全字段。因此消费语义必须在此路径之前阻断，metadata preservation 也不能只靠测试 fixture 恰好复制正确值。[Source: `rpg_engine/save.py`; `rpg_engine/db.py`]
- `delta_schema.py` 负责通用 JSON/entity shape 和 duplicate entity id 等基础规则，但不了解单物品消费语义；不要把 action-specific payload 字段硬编码进通用 schema，除非开发证据证明 resolver 无法安全拥有该合同。[Source: `rpg_engine/delta_schema.py`]
- Combat 已在 `validate_combat_delta()` / `validate_ammo_decrement()` 中管理 fired-ammo exact decrement；craft 使用不同的 `materials[]` 与 output payload contract。Story 1.9 应沿用 action owner 模式而不是用 `material_consumption_required` 全局匹配。[Source: `rpg_engine/actions/combat.py`; `rpg_engine/actions/craft.py`]

### 事务、authority 与 replay 约束

- `data/game.sqlite` 是 Save 当前事实权威；events JSONL、snapshots、cards、memory、registry、pending、preflight 与 audit 都是 projection/entry/evidence，不能覆盖事实。[Source: `docs/save-and-campaign-packages.md`]
- 普通玩家写入链保持 `player_turn -> pending action -> player_confirm -> validation + commit`。Pending 不是事实；非过期 validation failure 不应被消费 validator 擅自清除或改写。[Source: `docs/save-and-campaign-packages.md`; `_bmad-output/implementation-artifacts/1-3-player-confirm-validation-commit-gate.md`]
- Story 6.4 已完成 owner-validated confirmation replay。消费 fresh commit 成功后，可信 replay 应基于 durable SQLite/receipt evidence 返回既有结果，不能重新执行 fresh write 或二次扣量；low-level caller 仍不得自报 replay authority。[Source: Git commit `dff6e4dd0d2e2d0712d32089efdef4bef65b2d82`; `tests/test_pending_confirmation_replay.py`]
- Resolver validator 只能读取传入 connection 并返回 `ActionValidationResult`；不得写 SQLite、events JSONL、pending、backup、projection、registry 或正式数据，也不得让 AI/Skill/player text 成为事实或确认权威。[Source: `docs/architecture.md`; `docs/project-context.md`]
