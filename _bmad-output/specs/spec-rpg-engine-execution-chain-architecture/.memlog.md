- (direction by user) Create a bmad-spec after investigating RPG Engine architecture and execution chain; use the concluded investigation as the primary input.
- (decision) Spec target is a downstream architecture contract for RPG Engine execution-chain work, not a code implementation task.
- (constraint) Preserve the project boundary: AI proposes; Kernel verifies; Player confirms; Engine commits.
- (constraint) External AI remains low-trust; internal AI review cannot preview, validate, confirm, or commit gameplay facts.
- (constraint) SaveManager.player_turn may create pending action but cannot commit facts; SaveManager.player_confirm remains the ordinary player commit gate.
- (capability) CAP-1: Architecture consumers can identify and preserve the ordinary player-safe pending-confirm-commit chain.
- (capability) CAP-2: Architecture consumers can classify public and semi-public entry surfaces into player-safe, trusted low-level, maintenance/admin, platform, and prewarm categories.
- (capability) CAP-3: Architecture consumers can define an IntentCoordinator or equivalent boundary for candidate preparation, preflight identity, AI consensus, and proposal provenance without gaining write authority.
- (capability) CAP-4: Architecture consumers can state projection/outbox as post-commit repairable read-model and evidence surfaces, not fact authority.
- (capability) CAP-5: Downstream stories can derive verification gates for player profile access, pending confirmation, preflight identity and single-use behavior, and projection/outbox repairability.
- (non-goal) Do not implement code, migrations, or refactors in this spec.
- (non-goal) Do not reclassify preflight as a proposal cache, permission cache, or gameplay fact source.
- (assumption) The next primary downstream skill is bmad-architecture because gds-game-architecture requires a GDD and RPG Engine is a software kernel rather than a traditional game client.
- (question) Should legacy/admin CLI taxonomy be exhaustive before architecture, or can architecture define taxonomy first and leave exhaustive command inventory to a story?
