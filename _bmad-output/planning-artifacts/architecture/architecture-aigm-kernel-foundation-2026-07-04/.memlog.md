- (constraint) Inherit execution-chain spine as binding: ordinary player writes pass through SaveManager.player_turn pending state, SaveManager.player_confirm gate, GMRuntime.commit_turn validation/commit, and projection/outbox remains post-commit evidence.
- (constraint) PRD is final and validated Excellent; architecture must preserve AIGM Kernel foundation requirements and not re-open settled product decisions.
- (decision) Paradigm: local-first contract-centered AIGM kernel with thin surfaces, scenario-owned content, save-owned runtime facts, service-owned access contracts, and advisory AI assistants. Binds foundation implementation so two feature epics do not invent incompatible authority or ownership models.
- (decision) Scenario Package owns authored world settings, initial entities, initial relationships, progress definitions, rules, prompts, smoke tests, and gameplay scaffolding; Save Package owns runtime facts, relationship/progress changes, events, pending state, and projections; Kernel owns schema, import, validation, query, context assembly, visibility, and commit boundaries.
- (decision) AI outputs from external AI, internal intent AI, resident AI coordinator, and resident assistants are low-trust candidates or advisory records only; none can confirm, authorize hidden access, construct trusted deltas outside resolvers, bypass validation, or commit facts.
- (decision) Entity remains the shared identity anchor; Relationship and Progress Track/Clock are first-class contracts exposed through services/APIs even if v1 storage starts from existing relationship entity/details and clocks tables. This prevents callers from depending on ad hoc details_json or separate identity systems.
- (decision) Context Slice is the required AI/host context contract: context assembly must output inspectable scoped items with visibility, source/provenance, inclusion reason, budget evidence, and omitted/missing signals before rendering prompts.
- (decision) Resident AI v1 is a Resident AI Coordinator plus narrow assistants for intent recognition, context summarization, entity maintenance advice, progress management advice, and plot progression advice; coordinator schedules and records provenance but writes only advisory outputs.
- (decision) Generic gameplay variation is carried by Scenario Package capabilities, content types, rules, smoke tests, and extension hooks; Kernel core may add new generic primitives only via explicit architecture/story decision, not per-scenario forks.
- (decision) Latency policy is tunable but authority-preserving: player-facing AI assistance targets about 8s soft wait and about 15s hard timeout candidate; resident/background advisory may finish async in 30-60s and must not block commit.
- (decision) Required contract families are Scenario Package, Save Fact, Intent Candidate, Context Slice, Resident AI Advisory, Entity/Relationship/Progress Access, Validation/Commit, and Surface Authority; each must define inputs, outputs, errors, permissions, visibility, fact authority, and tests.
- (decision) Plot progression v1 is represented as plot progression signals in context and advisory outputs, not a mandatory storylet.yaml, scheduler, or linear director; future Storylet/Plot Beat structures remain optional scenario schema/context hints unless a later architecture binds them.
- (decision) Scenario diagnostics v1 must prioritize AI/fact/hidden breakage: manifest/schema parse, required roots, entity references, relationship endpoints, progress/clock completeness, visibility leakage risk, capabilities versus smoke tests, missing summaries and aliases.
- (constraint) Brownfield reality: current implementation already uses Python 3.11+, SQLite fact authority, entities as shared anchor, relationship content as relationship entities, clocks table for progress, context_builder/context collectors/audit for context, and AIIntentRouter for candidate arbitration. Ratify and refine rather than rewrite.
- (decision) Operations envelope remains local-first single-player/single-campaign v1: no cloud/distributed/public server architecture, no multiplayer coordination, no broad UI architecture; maintenance/admin surfaces remain gated and auditable.
